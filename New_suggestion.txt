AmiSSL 5.27 (OpenSSL 3.x) AROS x86_64 Port — Debugging Notes
============================================================
Date: June 15, 2026
Submitted by: AmiSSL porting team

== Key Finding ==

The AROS x86_64 kernel's PC32 relocation formula in
rom/dos/internalloadseg_elf.c (line 431-433) is CORRECT for all
critical code paths. No kernel changes are needed for AmiSSL.

The original formula:

    case R_X86_64_PLT32:
    case R_X86_64_PC32:
        *(ULONG *)p = s + rel->addend - (IPTR) p;
        break;

This formula is correct for PLT32 (addend=-4) and for PC32 with
addend=-4 (which covers all external symbol references like SysBase).
PC32 relocations with other addends (used in .eh_frame tables) are off
by +4, but this is harmless — .eh_frame tables are never read on AROS.

== Actual Issue Found ==

The OpenSSL 3.x provider system was NOT the cause of SSL_CTX_new()
hanging. The hang was in the EVP_PKEY_asn1_find_str() function called
from ssl_load_ciphers() during GOST/MAGMA/KUZNYECHIK algorithm checks.
This function searches the ASN1 method table and appears to hang on
AROS x86_64 for certain GOST-related lookups.

== Fix Applied (AmiSSL-side, no kernel changes) ==

1. Rebuilt OpenSSL with -mcmodel=large (eliminates PLT32 from 89K to 9)
2. SSL_CTX_new() uses a minimal initialization that avoids the
   provider system entirely for cipher/digest loading. Ciphers and
   digests are loaded via the legacy OBJ_NAME table using
   OPENSSL_INIT_ADD_ALL_CIPHERS/ADD_ALL_DIGESTS + EVP_get_cipherbyname/
   EVP_get_digestbyname. This completely bypasses the provider dispatch
   mechanism.
3. The GOST/MAGMA/KUZNYECHIK checks in ssl_load_ciphers are skipped
   (return 1 before them).
4. X509_STORE_new skips CRYPTO_new_ex_data (hangs on AROS).
5. CRYPTO_THREAD_lock_new returns real AROS SignalSemaphores.
6. All stores (evp_method, provider, namemap, property, DRBG) are
   initialized in context_init.

== Remaining Issues for Full Provider Support ==

If full OpenSSL 3.x provider support is needed on AROS x86_64,
the following need investigation:

1. EVP_PKEY_asn1_find_str() hangs for certain GOST key lookups
2. OSSL_PROVIDER_do_all() / ossl_provider_doall_activated() may hang
3. CRYPTO_new_ex_data() hangs when called through X509_STORE_new

These are likely related to how the AROS kernel handles function
pointer dispatch tables used by the OpenSSL provider framework,
not to the PC32/PLT32 relocation formulas which are correct.
